= Relation of kind `OneToMany`

== Domain story

This implementation guide applies whenever one entity instance owns a collection of child entity instances, and the relationship is managed through a `@ManyToOne` back-reference on the child side.
A bidirectional mapping is usually more efficient and idiomatic.

.Example
....
One owner can have many pets.
The inverse side of the relation is declared on entity `Owner` and mapped via the `owner` field in entity `Pet`.
The pets are read lazily, cascade-persisted with their owner, and automatically deleted when removed from the collection.
They are exposed only as lightweight item objects in API responses to avoid circular references.
....

== Default implementation

.Entity
[source,java,options="nowrap"]
----
@OneToMany(
        fetch = FetchType.LAZY, <1>
        mappedBy = "owner", <2>
        cascade = CascadeType.ALL, <3>
        orphanRemoval = true <4>
)
@Getter(value = AccessLevel.NONE) <5>
@JsonProperty(access = JsonProperty.Access.WRITE_ONLY) <6>
private Set<Pet> allPet; <7>
----

<1> The field is marked with `@OneToMany` with `LAZY` fetch type.
The collection is not loaded immediately; it's fetched only when accessed.
<2> The `mappedBy` attribute indicates this is the inverse side of the relationship.
It refers to the field name `owner` in the `Pet` entity.
<3> The `cascade = CascadeType.ALL` ensures all persistence operations (persist, merge, remove, etc.) cascade to the children.
<4> The `orphanRemoval = true` means that when a child entity is removed from the collection, it is automatically deleted from the database.
<5> The field is marked with `@Getter`.
This prevents the collection to be accessed via a getter method.
<6> The field is marked with `@JsonProperty`.
The collection is only used during deserialization, not in JSON responses.
<7> The field has the data type `Set<Pet>`.
Using a `Set` prevents duplicate children.

== Liquibase changeset

The relation does not require changes in the database on this side of the mapping.

== Relation initialization

The relation is initialized with an empty collection to avoid null pointer exceptions.
Use `LinkedHashSet` to maintain insertion order.

The `withId` method must copy the relation so that cloned instances remain semantically equal.

== Relation manipulation

== Relation verification

The relation participates in the `isEqual` method comparison.
Use `Objects#equals`.

Do not use relations in the `verify` operation.

== Relation serialization

The relation is by default *not* part of the JSON structure.

The relation can be added to the JSON structure only with item classes.
This prevents circular references and reduces payload size.
Be aware of the potential impact on performance.
Use this approach only for relations with a small number of elements.

.Items
[source,java,options="nowrap"]
----
@Override <1>
protected void extraJson(@NonNull final Map<String, Object> allExtra) {
    allExtra.put("allPetItem", allPet.stream() <2>
        .map(PetItem::fromValue)
        .collect(Collectors.toSet()));
}
----

<1> Callback from the base operation with `@JsonAnyGetter` annotation.
<2> All child entities are converted to a simple state transfer object.
